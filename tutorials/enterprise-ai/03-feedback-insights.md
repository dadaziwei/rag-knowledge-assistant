# 03 知识缺口分析与反馈洞察

## 这个功能解决什么问题

上一轮我们已经把“有帮助 / 待改进”反馈采集进系统了，但如果这些反馈只是躺在数据库里，团队还是很难真正拿它做优化。

企业更关心的是：

- 最近哪些问题最容易让用户不满意
- 哪些知识库被命中后仍然经常收到“待改进”
- 当前整体有帮助率大概是多少

所以这一轮的目标是：

```text
把离散反馈
  ↓
聚合成可以直接阅读的反馈洞察
  ↓
帮助团队发现知识缺口和优化优先级
```

## 为什么企业需要

企业内部知识助手最终不是比谁“回答得像人”，而是比谁能持续提升效果。

如果没有反馈洞察，团队只能凭感觉改：

- 可能一直在优化不重要的问题
- 可能不知道是知识库内容缺失，还是检索命中不准
- 可能有大量差评，但没人能及时发现

所以一个更像企业系统的做法是：

1. 先采集用户反馈
2. 再聚合成简明统计
3. 用统计定位知识缺口
4. 再决定补文档、调检索还是改提示词

## 这一轮改了什么

### 后端

- 在 `backend/app/db/mongo.py` 中新增 `get_feedback_insights`
- 遍历当前用户所有对话，聚合：
  - 总反馈数
  - 有帮助数量
  - 待改进数量
  - 有帮助率
  - 待改进最多的知识库
  - 最近被标记为“待改进”的问题
- 在 `backend/app/api/endpoints/chat.py` 中新增反馈洞察接口

接口返回的是一个轻量摘要，而不是复杂报表，便于后续继续扩展。

### 前端

- 在 `frontend/src/app/[locale]/ai-chat/page.tsx` 中新增反馈洞察拉取逻辑
- 在 `frontend/src/components/AiChat/LeftSidebar.tsx` 中新增“反馈洞察”卡片
- 卡片展示：
  - 累计反馈
  - 有帮助率
  - 有帮助 / 待改进数量
  - 待改进最多的知识库
  - 最近待改进问题

## 怎么验证

### 功能验证

1. 先对几条回答点“有帮助”或“待改进”
2. 返回左侧边栏
3. 检查是否出现“反馈洞察”卡片
4. 检查总反馈数、比例和知识库分布是否更新
5. 对另一条回答再次点反馈
6. 检查卡片是否刷新，最近待改进问题是否变化

### 契约验证

运行：

```powershell
python -m unittest tests.test_resume_rag_upgrade
```

确认新增的 feedback insights 契约测试通过。

### 安全验证

提交前执行敏感信息扫描，确认没有把真实密钥、token 或默认凭据带进仓库：

```powershell
rg -n -uu --hidden --glob '!**/.git/**' "replace-with-your-|real api key|real token|default credentials" .
```

## 这一步的价值

这一轮仍然是轻量版，但它已经把“反馈采集”升级成了“反馈可用”：

- 团队可以快速知道系统效果大致如何
- 可以知道哪几个知识库最值得优先补内容
- 可以知道最近哪些问题更容易让用户不满意
- 后续继续往上做知识缺口报告、离线评测和检索优化时，就有了稳定入口

## 简历怎么写

> 基于用户问答反馈构建轻量知识缺口分析能力，聚合有帮助率、待改进问题与知识库分布，在对话侧提供可视化反馈洞察，为知识库补全和 RAG 效果优化提供数据依据。

## 面试怎么讲

> 我没有把反馈只停留在点赞点踩本身，而是继续做了一层反馈洞察。后端会聚合用户历史对话里的反馈结果，统计总反馈数、有帮助率、被标记为待改进最多的知识库，以及最近需要关注的问题。前端在侧边栏放了一个很轻量的洞察卡，用户和维护者都能直观看到当前系统的效果趋势。这样系统就不只是“能回答”，而是开始具备“发现问题并持续改进”的能力。
